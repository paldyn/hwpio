# 수행계획서 — Task M100-1205: HWPX 문단 borderFill NONE side 렌더링 정합성

## 1. 이슈

- GitHub Issue: [#1205](https://github.com/edwardkim/rhwp/issues/1205)
- 제목: `HWPX: 문단 borderFill의 NONE side가 무시되어 연결 테두리가 사각형으로 렌더링됨`
- 대상 마일스톤: M100 / v1.0.0 편집 기반
- 브랜치: `issue-1205-para-border-none-side`
- 기준 커밋: `5137c07f`

- 작성일: 2026-06-03

## 2. 문제 요약

HWPX 문서의 문단 테두리에서 `borderFill`의 좌우 side가 `NONE`인데도, rhwp 렌더링 결과에서는 연결 문단 border가 4면 사각형처럼 보인다.

이슈 본문의 재현 샘플에서는 `borderFill id=23`이 다음 의미를 가진다.

| side | type | 기대 동작 |
|------|------|-----------|
| left | `NONE` | 렌더링하지 않음 |
| right | `NONE` | 렌더링하지 않음 |
| top | `SOLID` | 상단 가로선 렌더링 |
| bottom | `SOLID` | 하단 가로선 렌더링 |

따라서 문단 border를 연결해 병합하더라도 side별 `BorderLineType`은 렌더링까지 보존되어야 한다.

## 3. 현재 코드 관찰

코드 수정 없이 확인한 현재 구조는 다음과 같다.

- `src/parser/hwpx/header.rs`
  - `leftBorder`, `rightBorder`, `topBorder`, `bottomBorder`를 side별로 파싱한다.
  - `type="NONE"`은 `BorderLineType::None`으로 매핑된다.
  - 따라서 현재 문제의 1차 원인은 파서보다 렌더러 경로일 가능성이 높다.
- `src/renderer/layout.rs`
  - 문단 border group의 stroke signature가 top border 중심으로 계산된다.
  - 일반 경로에서는 `RectangleNode`에 stroke를 주어 4면을 한 번에 그린다.
  - partial/cross-column 경로에서는 `LineNode`로 분해하지만 좌우 수직선을 항상 렌더링한다.
  - 이 구조에서는 `left/right = NONE`인 borderFill도 top/bottom이 보이면 사각형처럼 렌더링될 수 있다.
- `src/renderer/layout/border_rendering.rs`
  - 표 테두리 계열은 side별 `BorderLine`을 기반으로 line을 생성하는 유틸리티를 이미 가진다.
  - 문단 border도 같은 방향으로 side별 가시성을 반영할 여지가 있다.

## 4. 목표

문단 border 렌더링에서 `borderFill`의 side별 `BorderLineType`을 존중한다.

완료 후 기대 동작:

1. `leftBorder/rightBorder = NONE`이면 연결 문단 border 그룹에서도 좌우 수직선을 그리지 않는다.
2. `topBorder/bottomBorder = SOLID`이면 필요한 가로선은 유지한다.
3. `connect=1`에 따른 문단 border 병합 동작은 유지한다.
4. 기존 4면 문단 border, cross-column/cross-page border 회귀를 만들지 않는다.

## 5. 작업 범위

이번 이슈의 범위:

- 문단 border group 렌더링에서 side별 border visibility를 반영한다.
- 단일 `RectangleNode` stroke로 4면을 그리는 경로를 점검하고, 필요한 경우 fill과 stroke를 분리한다.
- partial/cross-column 경로에서 좌우 수직선을 항상 그리는 로직을 side별 조건으로 바꾼다.
- `BorderLineType::None`인 side는 width/color 값이 있어도 비가시 side로 처리한다.
- 회귀 테스트를 추가해 left/right `NONE`, top/bottom `SOLID` 조합이 사각형으로 렌더되지 않음을 고정한다.

이번 이슈의 비범위:

- HWPX `borderFill` 파싱 구조 전면 변경.
- 표 셀 테두리 렌더링 알고리즘 변경.
- 문단 border spacing, ignoreMargin, connect 규칙의 의미 확장.
- 대상 샘플 파일명이나 문단 index에 의존하는 특수 분기.
- #1196 맞쪽 여백, #1197 z-order, #1201 바탕쪽 관련 동작 변경.

## 7. 구현 방향 후보

### 후보 A: 문단 border stroke를 항상 side별 `LineNode`로 분해

문단 border group의 fill은 `RectangleNode`로 유지하고, stroke는 left/right/top/bottom 각각의 `BorderLine`을 확인해 `LineNode`로 생성한다.

장점:

- `NONE` side 표현이 명확하다.
- 일반 경로와 partial 경로의 렌더링 계약을 통일할 수 있다.

위험:

- 기존 단일 `RectangleNode` stroke 기반 테스트나 SVG 구조 기대가 달라질 수 있다.
- 4면 박스가 필요한 기존 문단 border에서 line join 표현이 달라질 수 있다.

### 후보 B: `RectangleNode` 유지 조건을 제한하고 혼합 side만 분해

4면이 모두 같은 visible stroke인 경우에는 기존 `RectangleNode` 경로를 유지하고, side 중 하나라도 `NONE`이거나 서로 다르면 `LineNode`로 분해한다.

장점:

- 기존 4면 사각형 border 출력 구조를 최대한 유지한다.
- 이번 이슈의 직접 결함만 좁게 고칠 수 있다.

위험:

- 일반 경로와 partial 경로의 조건 분기가 복잡해질 수 있다.
- side별 line type/width/color가 서로 다른 문단 border까지 완전히 처리하려면 추가 확장이 필요하다.

## 8. 기본안

기본안은 후보 B로 진행한다.

```text
1. 문단 border group의 borderFill을 side별로 조회한다.
2. 4면이 모두 visible이고 동일 stroke일 때만 기존 Rectangle stroke 경로를 유지한다.
3. 일부 side가 NONE이거나 partial 경로가 필요한 경우 fill과 stroke를 분리한다.
4. stroke는 visible side만 LineNode로 생성한다.
5. partial_start/partial_end에서는 기존 top/bottom skip 의미를 유지하되, left/right도 borderFill side visibility를 확인한다.
```

이 방식은 기존 4면 문단 border의 렌더 구조를 가능한 보존하면서 #1205의 `NONE` side 오렌더를 막는 것을 우선한다.
