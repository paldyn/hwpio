### 1.4 글상자 LIST_HEADER raw byte 비교

```
ORACLE (size=33):
  02 00 00 00          paraCount = 2
  20 00 00 00          list_attr = 0x00000020
  1b 01 1b 01 1b 01 1b 01    margin_left/right/top/bottom (4×i16)
  ea 30 00 00          max_width = 0x30ea = 12522
  00 00 00 00 00 00 00 00    추가 8 byte (zero)
  00 00 00 00 00        추가 5 byte (zero)
  
rhwp (size=20):
  02 00 00 00 20 00 00 00 1b 01 1b 01 1b 01 1b 01 ea 30 00 00
                                                            ↑ 여기서 끝 (13 byte 누락)
```

→ 정답지의 글상자 LIST_HEADER 는 **마지막 13 byte (모두 zero) 추가** 존재.

### 1.5 가설

**한컴편집기 가설**: 글상자 (TextBox) 의 LIST_HEADER 가 한컴이 기대하는 형식보다 짧으면 한컴은 이를 본문 list 의 일부로 해석하고, 안에 추가 paragraph (각주) 생성 시 **본문 다단계 목록 번호 (1.1.1.1.1.1.)** 자동 부여. 글상자 안 paragraph 들이 정상 글상자 list 로 인식 안 됨.

추가 13 byte 는 한컴 내부의 **paragraph numbering / level / list_id 메타** 정보일 가능성. rhwp 가 저장 시 누락 → 한컴이 default 처리 → 다단계 목록 부여.

## 2. 처리 방향

**작업지시자 지시**: "이번 사안은 정답지와 저장 버전과 차이를 통해 추론" — Stage 4-pivot 의 HWP IR oracle 방식 동일 적용.

`feedback_hancom_compat_specific_over_general` 정합 — case-specific 한컴 contract 보존 (raw 13 byte 추가 + zero padding).

## 3. 코드 영역 매트릭스

| 영역 | 현재 | 본 task 변경 |
|------|------|------|
| `src/serializer/control.rs::serialize_text_box_if_present` (line 1434) | LIST_HEADER 20 byte (paraCount + list_attr + margin × 4 + max_width + raw_list_header_extra) | **+13 byte 추가** (raw_list_header_extra 가 비어있을 때 한컴 default 13 byte zero) |
| `src/parser/control.rs` 의 LIST_HEADER 파싱 | raw_list_header_extra 에 33 byte LIST_HEADER 의 추가 부분 보존 | 확인 필요 — 이미 보존하면 본 task 는 serializer 만 |
| `src/model/shape.rs` TextBox.raw_list_header_extra | 보존 영역 존재 | 정합 확인 |

설계: 기존 `raw_list_header_extra` 패턴 활용. HWPX 출처는 raw 부재 → **한컴 default 13 byte zero 적용**.
