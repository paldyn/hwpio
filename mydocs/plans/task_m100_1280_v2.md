# Task #1280 (v2) 수행 계획서 — 글상자 위 이미지 배치(plane)/클릭 우선순위

## 이슈

- GitHub Issue: #1280 (후속, `_v2`)
- 원 제목: rhwp-studio: 삽입한 글상자가 text_box 없는 Rectangle로 생성되어 커서 진입·타이핑·붙여넣기 실패 (글상자 없음)
- v2 범위: **삽입 글상자 생성 결함의 후속** — 글상자가 한컴과 다른 배치(plane)/floating 속성으로
  생성되어 위에 올린 이미지가 글상자를 가리고, 겹침 클릭이 어긋난다.
- 처리 주체: #1280과 동일 (마일스톤·orders·이슈 클로즈는 메인테이너 영역).

## 배경

#1280 본편은 삽입 글상자가 `text_box` 없는 Rectangle로 생성되던 것을 `shapeType:'textbox'`로
고쳐 **텍스트 입력**을 복구했다(stage1~3 완료). 그 과정에서 글상자를 **인라인(treat_as_char=true,
offset=0, 커서 위치)** 으로 배치하도록 확정했다.

그런데 후속 사용에서 새로운 결함이 드러났다: **글상자 위에 이미지를 삽입하면**

1. **렌더링**: 이미지가 글상자 *위*로 그려진다 (한컴은 이미지가 글상자 *뒤*로 감).
2. **히트테스트**: 위에 보이는 이미지를 클릭해도 선택되지 않고 밑의 글상자가 선택된다
   (보이는 것 ≠ 클릭되는 것, **WYSIWYG 불일치**).

## 재현 대상

1. rhwp-studio 새 문서
2. 입력 → 글상자 삽입 → 드래그로 글상자 영역 지정
3. 입력 → 그림 → 글상자 위에 겹치도록 이미지 배치
4. → 이미지가 글상자를 가림(한컴과 반대). 겹침 영역 클릭 시 선택 대상이 한컴과 어긋남.

## 근본 원인 (실측 + 코드 검증 완료 — ⚠️ z-order 아님, plane 문제)

한컴 권위 샘플 `samples/textbox-under-image.hwp` 를 `rhwp dump` 한 결과:

- `[2] 사각형(글상자)`: **배치=글앞으로(InFrontOfText)**, 글자처럼=false(floating), **z=0**
- `[3] 그림(이미지)`: **배치=어울림(Square)**, 글자처럼=false, **z=1**

→ 한컴에서 이미지가 글상자 뒤로 가는 건 **z-order 때문이 아니다**(z는 오히려 이미지1 > 글상자0).
**글상자가 `글앞으로(InFrontOfText)` plane(3)** 이라 어울림 이미지(plane 2) 위에 그려지기 때문이다.
렌더 정렬키 `(plane, z_order, stable_index)` 에서 plane이 상위라 이김.

1. **렌더링 불일치(진짜 원인)** — rhwp-studio가 만드는 글상자는 한컴과 정반대:
   - `text_wrap=Square`(frontend가 `textWrap` 미전달 → `wasm_api.rs:2867` 기본 "Square" →
     `common.text_wrap` `object_ops.rs:3795`)
   - `treat_as_char=true`(인라인; `wasm_api.rs:2865` 기본, `object_ops.rs:3782`)
   - `attr=0x0A0210`(Square 비트; `object_ops.rs:3676`)
   → 글상자가 plane 2(어울림)+인라인이라 floating 이미지(plane 2)에 깔린다. #1280이 정한
   인라인 배치가 이 plane 결함의 직접 원인.
2. **히트테스트 불일치** — `findPictureAtClick` 1차 패스가 `layout.controls`(렌더 트리를
   `(plane, z_order, stable_index)` 오름차순으로 펼친 배열)에서 **첫 bbox 적중**을 반환한다
   (`input-handler-picture.ts:158-161`). 즉 **최하단** 개체를 선택한다. 주석에도 "z-order 미지원".

## 스펙 정합 (한글문서파일형식 5.0 rev1.3)

- z-order = 개체별 `INT32`(표 69, 1452행), 글상자=사각형 그리기개체+글상자 텍스트(표 89/90)로
  그림과 동일 공통속성 공유(표 67/81). plane=BehindText/InFrontOfText(표 70 bit21~23).
- 단 라인 1508 errata대로 HWP5 저장 wrap값은 spec bit와 다름 → raw bit 아닌 `TextWrap` enum 기준.
- 스펙은 파일 포맷만 규정(삽입 정책은 편집기 동작)이므로 **한컴 샘플 실측으로 정책 확정**(위).

## 수정 방향

**작업지시자 결정: 한컴 동일(이미지가 글상자 뒤로).** 두 축으로 수정한다.

1. **히트테스트 "클릭 = 최상단 개체 선택"**(공통, 어느 배치든 필요): `findPictureAtClick` 1차 패스를
   첫-적중-반환 → 적중 후보 중 `(plane, zOrder, stableIndex)` 최대값 선택으로 교체. 이를 위해
   Rust 레이아웃 쿼리에서 컨트롤별 `zOrder/plane/stableIndex/wrap`을 노출한다.
   - #1171 nested-picture 우선(Pass 0), #516 BehindText 폴백(Pass 2)은 **그대로 보존**.
2. **글상자 삽입 기본값을 한컴 정답값으로 교정**: `InFrontOfText` + `floating(treat_as_char=false)`.
   - #1280이 정한 인라인(treat_as_char=true)을 floating으로 되돌린다.
   - 회귀 위험 완화 통찰: 기존 HWP/HWPX 글상자는 이미 floating이고 텍스트 입력이 정상 동작한다.
     floating 복귀는 **삽입 글상자 = 로드된 글상자**로 정합시키는 방향이라 #1280 텍스트 입력
     회귀 위험이 낮다.

## 진행 계획 (5단계)

구현 계획서(`task_m100_1280_v2_impl.md`)에 상세화. 각 단계 독립 커밋·테스트 동반.

1. **Stage 1** — `rendering.rs` `collect_controls`에 `zOrder/plane/stableIndex/wrap` 노출 + Rust 단위.
2. **Stage 2** — `types.ts` `ControlLayoutItem` 필드 확장(동작 무변화) + 프런트 단위.
3. **Stage 3** — `findPictureAtClick` 1차 패스 최상단 선택 교체 + e2e(`topmost-hittest.test.mjs`).
4. **Stage 4** — 렌더/히트테스트 정합 회귀 e2e 확장(BehindText/중첩/line/다중구역).
5. **Stage 5** — 글상자 삽입 기본값 floating+InFrontOfText 교정(프런트 명시 전달 + attr 비트 정합)
   + floating 위치 산정 + Rust/e2e 테스트 + #1280 텍스트 입력 회귀 확인.

## 완료 기준

1. 글상자 위에 이미지를 삽입하면 **이미지가 글상자 뒤**로 가고(한컴 동일), 겹침 클릭 시 **위에 보이는
   개체(글상자)가 선택**된다.
2. 삽입 글상자가 `InFrontOfText` + `treat_as_char=false`(floating)로 생성되고, attr 비트와
   `common.text_wrap`이 정합한다(HWP5 round-trip 무손상).
3. 히트테스트가 겹침 시 `(plane, zOrder, stableIndex)` 최상단 개체를 선택한다.
4. #1280 본편 회귀(글상자 텍스트 입력/붙여넣기), #1171/#516 회귀가 없다.
5. 신규 e2e + Rust 단위 테스트 통과.
