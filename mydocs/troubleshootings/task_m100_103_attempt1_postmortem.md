4. `layout_column_shapes_pass`에서 앵커 문단 y + `vertical_offset` 위치에 독립 배치

## 시도 순서와 실패 흐름

22곳의 `match PageItem` 에 `FloatingTable` arm 추가. 기능 변경 없음. 회귀 없음.

### 2단계 — 분류 변경 (예상대로 실패 발생)

비-TAC Para-relative 표가 `FloatingTable`로 분류되기 시작.
- 3개 테스트 실패: `test_page12_enter_table_placement`, `test_task105_nested_table_path_api`, `test_task77_image_cell_no_intra_row_split`

### 3단계 — dead code 제거 (겉으로는 성공)

`effective_table_height`, `is_independent_float` 등 Para-relative 전용 보정 패치 29줄 제거. 2단계와 동일한 실패 스코프 유지. "정상" 판정.

**판단 오류 심화**: 제거된 패치들은 **원래 HWP 엔진의 혼합 모델(in-flow + 독립 y)을 1패스로 흉내내는 정교한 보정**이었다. 이걸 "dead code"로 취급한 것은 "2패스 설계가 올바르다"는 전제 하에서만 성립. 그 전제를 증명하지 않은 채 제거했다.

### 4단계 — 2패스 배치 구현 (결정적 문제 노출)

`layout_column_shapes_pass` 확장, `layout_table`을 `col_node` 하위에 배치.
결과:
- 페이지가 1페이지로 축소 (모든 문단이 1페이지에 쌓임)
- `LAYOUT_OVERFLOW` 다수 발생 (FloatingTable, 그 아래 본문 전부)
- `typeset=2` vs `paginator=1` 불일치
- **표는 배치되지만 본문이 표 공간을 침범**하여 원래 버그(#157)와 시각적으로 유사한 결과

**전략 수정 시도**: `shape_reserved`에 FloatingTable 포함시키기 → 해당 함수가 `vert_rel_to=Para`를 **의도적으로 제외**하고 있음을 발견 (코멘트: "이중 밀림 방지").

**두 번째 수정 시도**: 1패스에서 FloatingTable을 만나면 `y_offset`을 `anchor_y + v_off + table_h`까지 점프. 결과: 페이지네이션 엔진은 여전히 높이 누적 안 하므로 2페이지로 분할되지 않음. 페이지네이션 엔진에도 동일 로직을 넣어야 함 → **1패스/2패스 이원화 구조 전체를 손대는 상황**.

**여기서 중단 결정**: 작업지시자의 지적 — "코드에서 억지로 이 엣지케이스를 통과시키려고 무리하는 것 아닌가요?"

## 잘못된 가정 (핵심)

### 가정 1: HWP wrap=위아래 표는 CSS float와 동등하다

**틀림**. CSS float는:
- 요소를 인라인 흐름에서 빼내어 좌/우로 밀어붙임
- 뒤따르는 인라인 컨텐츠가 float 옆으로 흐름

HWP `wrap=위아래` 표는:
- **세로 공간을 완전히 점유** (옆으로 흐름 없음)
- 앵커 문단 기준 독립 y에 배치되지만 **본문은 표 하단까지 점프**
- 즉 "본문에 보이지 않는 수직 예약 공간"처럼 동작

이 두 모델은 완전히 다르다. float 모델에 끼워맞추려 하면 이중 밀림, 불일치, overflow가 반복된다.

### 가정 2: Shape 2패스 구조를 표에 그대로 확장 가능

**틀림**. Shape 2패스는 **paper-based 개체**(`vert_rel_to=Paper/Page`)와 `InFrontOfText/BehindText` 표 전용이다. 이들은 진정한 out-of-flow (본문 흐름에 아무 영향 없음).

`wrap=위아래 + vert_rel_to=Para` 표는 **본문 흐름에 영향을 준다** (뒤 본문을 밀어내야 함). Shape 2패스의 전제와 근본적으로 다르다.

### 가정 3: 제거한 패치들은 "임시 우회 코드"였다

**틀림**. 3단계에서 제거한 `effective_table_height` 등은 **혼합 모델의 핵심 로직**이었다:
- Para-relative 표가 in-flow로 높이 기여 하되
- 피트 판단은 문단 시작 y 기준 (같은 앵커의 여러 표가 서로 다른 vert offset을 가질 때 각자 독립 판단)
- 이 패치가 없으면 같은 문단에 TAC 표 + 비-TAC 표가 공존하는 케이스 (#103 원 재현 케이스) 자체가 성립하지 않는다

**"patch"라는 단어에 속은 것**. 실제로는 혼합 모델의 설계 포인트였다.

### 가정 4: "중간 상태 실패"는 4단계가 복구한다

**검증하지 않은 채로 진행**. 2단계에서 깨진 3개 테스트 중 어느 것이 "표 배치만 복구되면 pass"인지, 어느 것이 "설계 가정 자체가 맞지 않아 pass할 수 없음"인지 구분하지 않았다. 결과적으로 4단계에서 배치를 구현해도 `LAYOUT_OVERFLOW`가 원래의 버그를 재현하는 상황에 도달했다.

### 가정 5: 1패스에서 FloatingTable을 밀기로 고치면 된다

구현 종반에 꺼낸 카드. 이 시점에서 **설계가 2패스 순수성을 유지 못한다는 결정적 증거**였는데, 그것을 **수정 아이디어로 소화**하려 함. 페이지네이션 엔진까지 수정하려 확장. 이것이 작업지시자 경고를 받은 시점.

## 놓친 신호들

- **2단계 직후**: `TYPESET_VERIFY: paginator vs typeset` 불일치 경고 — 이게 "두 엔진이 같은 레이아웃 모델을 공유하지 않는다"는 구조적 신호였는데 "4단계에서 해결될 것"으로 치부
- **3단계 직후**: dead code 제거 후 실패 스코프 불변 — "제거가 안전하다"로 해석했지만, 실제로는 **이미 2단계에서 기능이 깨져 있어서 추가 회귀가 드러날 기회 자체가 없었던 것**
- **4단계 초반**: `shape_reserved`가 Para-relative를 **명시적으로** 제외 — 이 코멘트는 "현재 구조에서는 이 케이스를 다른 방식으로 처리해야 함"이라는 설계 메모인데 무시하고 우회
- **4단계 중반**: 페이지 수 축소 + 본문 침범 증상이 **원래 버그와 유사** — 이 시점이 "가정 자체가 틀렸다"는 강력한 신호. 그런데 "1패스 밀기를 넣으면 해결"이라는 새 수정으로 회피

1차 시도는 **"#103과 #157이 같은 원인"이라는 타당한 판단에서 출발했으나, 구현 전략(2패스 확장)을 검증하지 않고 진행**하여 실패했다. 실패 자체가 프로젝트의 손실이 아니라, 실패 시 정직하게 인정하지 않고 우회 수정을 누적하는 것이 더 큰 손실이다. 이번엔 작업지시자의 개입으로 중단 지점을 찾았다.

재도전은 반드시 **구현 전에 모델 확정 → 검증**을 거친다.
