# Task M100 #661 수행계획서

## 타이틀

rhwp-studio 텍스트 드래그 선택 중 커서와 스크롤 위치가 튀는 현상 정정

## 배경

Issue #661은 `rhwp-studio` web 편집 화면에서 텍스트를 드래그 선택할 때 커서 또는 페이지 스크롤 위치가 순간적으로 위아래로 튀는 문제이다.

이 문제는 Issue #658의 선택 하이라이트 overflow와 별도 결함이다. #658의 본질은 selection rect가 실제 텍스트/페이지 영역 밖으로 확장되는 문제이고, #661은 드래그 중 `hitTest`, caret rect, pageIndex, auto-scroll 경로가 서로 영향을 주면서 스크롤 위치가 불안정해지는 문제로 본다.

## 현상 요약

드래그 선택 중 포인터가 편집 영역 상단 또는 하단 auto-scroll 영역에 닿지 않았는데도 `scrollTop`이 순간적으로 바뀌거나, caret이 현재 드래그 위치와 다른 페이지/줄로 튀어 보인다.

이전에 Vite 로그에서 다음 경고가 반복 관찰되었다.

```text
[CursorState] 캐럿 페이지 불일치 (WASM=0, hitTest=1) -> hitTest 폴백
```

이는 `hitTest`가 반환한 `cursorRect.pageIndex`와 `CursorState.updateRect()`가 WASM `getCursorRect*`로 재계산한 `pageIndex`가 다를 수 있음을 의미한다.

## 의심 원인

### 1. rAF 내부에서 원본 MouseEvent 재해석

`rhwp-studio/src/engine/input-handler-mouse.ts`의 드래그 처리 경로는 `mousemove` 시점의 이벤트 객체를 rAF 콜백에서 뒤늦게 다시 해석한다.

현재 흐름:

```ts
if (this.isDragging) {
  if (this.dragRafId) return;
  this.dragRafId = requestAnimationFrame(() => {
    this.dragRafId = 0;
    if (!this.isDragging) return;
    const hit = this.hitTestFromEvent(e);
    if (hit && hit.paragraphIndex < 0xFFFFFF00) {
      this.cursor.moveTo(hit);
      this.updateCaretDuringDrag();
    }
  });
  return;
}
```

`clientX/clientY`는 화면 좌표다. rAF 실행 전후에 `scrollTop`이 바뀌면 같은 좌표가 다른 문서 좌표로 해석될 수 있다.

### 2. 드래그 중 caret 기준 자동 스크롤

#664 기준 `updateCaretDuringDrag()`는 드래그 중에도 `scrollCaretIntoView(rect)`를 호출한다.

일반 커서 이동에서는 적절하지만, 드래그 선택 중에는 포인터가 컨테이너 상/하단 auto-scroll 영역에 접근했을 때만 스크롤해야 한다. caret rect 기준 자동 스크롤은 pageIndex 불일치나 재계산 흔들림이 있을 때 `scrollTop` 피드백 루프를 만들 수 있다.

### 3. CursorState pageIndex 불일치 폴백

`rhwp-studio/src/engine/cursor.ts`는 WASM rect와 `hitTest.cursorRect`의 `pageIndex`가 다르면 hitTest rect로 폴백한다.

이 폴백 자체는 방어 코드지만, 드래그 중 반복 발생하면 caret 표시와 auto-scroll 판단이 매 프레임 다른 기준을 사용할 수 있다.

### 4. Rust getCursorRect*의 page hint 부재

`src/document_core/queries/cursor_rect.rs`의 `get_cursor_rect_native()`와 `get_cursor_rect_in_cell_native()`는 `find_pages_for_paragraph()` 결과를 앞에서부터 순회한다.

동일 문단 또는 표가 여러 페이지 후보에 걸리는 경우, 현재 마우스가 위치한 pageIndex가 아니라 앞쪽 페이지의 rect를 먼저 반환할 수 있다. 특히 셀/표/부분 문단이 섞이는 경우 page hint 없이 정확한 후보를 고르기 어렵다.

## 목표

드래그 선택 중 커서와 스크롤 위치가 포인터 입력에 안정적으로 대응하도록 정정한다.

완료 기준:

- 포인터가 편집 영역 상/하단 auto-scroll margin에 들어가지 않으면 드래그 중 `scrollTop`이 임의로 변하지 않는다.
- 드래그 중 `hitTest` 좌표가 rAF 지연과 스크롤 변화로 재해석되지 않는다.
- `CursorState` pageIndex 불일치 경고가 재현 시나리오에서 반복되지 않거나, 반복되더라도 스크롤 점프를 유발하지 않는다.
- `samples/exam_social.hwp` 2쪽 하단 좌측 박스 드래그 선택이 안정적으로 유지된다.
- #658의 selection rect overflow 정정과 충돌하지 않는다.

## 작업 범위

### 포함

- `rhwp-studio/src/engine/input-handler-mouse.ts`
  - 드래그 중 rAF 입력 snapshot 또는 hit snapshot 도입
  - 필요 시 명시적 pointer-edge auto-scroll 경로 분리

- `rhwp-studio/src/engine/input-handler.ts`
  - 드래그 중 caret 갱신 경로에서 caret 기준 auto-scroll 제거 또는 조건화
  - auto-scroll을 포인터 위치 기반으로 제한

- `rhwp-studio/src/engine/cursor.ts`
  - 드래그 중 `hitTest.cursorRect` 우선 사용 가능성 검토
  - pageIndex mismatch 경고/폴백이 스크롤 판단에 주는 영향 차단

- `src/document_core/queries/cursor_rect.rs`
  - 필요 시 page hint 기반 `getCursorRect*` 보조 API 검토
  - 단, 프론트엔드 경로만으로 안정화 가능하면 Rust API 변경은 뒤 단계로 미룬다

- `rhwp-studio/e2e/`
  - 가능하면 `exam_social.hwp` 기반 드래그 선택 진단 스크립트 또는 회귀 테스트 추가

## 진행 전략

1. 최신 `devel`과 PR #664 변경분 기준으로 드래그 경로를 대조한다.
