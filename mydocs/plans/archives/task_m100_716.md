## 1. 문제 정의

### 증상

`samples/20250130-hongbo.hwp` page 1 마지막 줄 (pi=15 line 2) 이 컬럼 하단을 +20.1px 초과하여 cropping 발생.

```
LAYOUT_OVERFLOW_DRAW: section=0 pi=15 line=2 y=1048.2 col_bottom=1028.0 overflow=20.1px
LAYOUT_OVERFLOW: page=0, col=0, para=15, type=PartialParagraph, y=1059.4, bottom=1028.0, overflow=31.3px
```

### 베이스라인 (integration/3pr-stack 기준)

VPOS_CORR 디버그 트레이스:
- pi=2 호출됨 (lazy, applied=false)
- **pi=3 호출 누락**
- pi=4 부터 재호출되나 모두 `applied=false`

→ pi=2 (TAC 표) 직후부터 layout y_offset 이 vpos 인코딩 예측보다 forward drift, VPOS_CORR 가 회수하지 못한 채 페이지 끝까지 누적 → pi=15 line 2 가 컬럼 하단 초과.

## 2. Root cause (사전 정적 분석)

### 본질: 음수 line_spacing 미반영 → drift 누적

이슈 본문은 "pi=2 → pi=3 진입 시 layout y_offset 시프트" 로 가설을 한정했으나, 분석 결과 **본질은 pi=2 단독이 아니라 음수 ls(line_spacing<0) 를 layout advance 가 무시하는 일반 패턴**.

페이지 1 의 첫 6개 항목별 실측 drift:

| pi | 종류 | lh (HU/px) | ls (HU/px) | HWP delta (lh+ls) | layout advance | drift |
|----|------|-----------|-----------|-------------------|----------------|-------|
| 0 | TAC table 호스트 | 3560 / 47.5 | -600 / -8 | 39.5px | ~47.5px (margin+table_h+margin) | **+8.0** |
| 1 | empty para | 1500 / 20 | -600 / -8 | 12.0px | 20.0px | **+8.0** |
| 2 | TAC table 호스트 | 3148 / 42 | -900 / -12 | 30.0px | ~42px | **+12.0** |
| 3 | empty para | 1500 / 20 | -900 / -12 | 8.0px | 20.0px | **+12.0** |
| 4 | empty para | 300 / 4 | +180 / +2.4 | 6.4px | 4.0px | -2.4 |

→ pi=5 진입 시 누적 drift +24.8px. 이후 빈 문단(pi=6,8,10,12,14)이 추가 누적 → pi=15 line 2 에서 +20.1px 초과 cropping.

### VPOS_CORR 보정 실패의 4중 가드

`src/renderer/layout.rs:1402-1555`:

1. **line 1402** `if !shape_jumped && !prev_tac_seg_applied` — TAC 표 직후 진입 차단 (pi=2→pi=3 trigger)
2. **line 1488-1494** `if lazy_base < 0 { (prev_vpos_end, false) }` — y_offset 이 vpos 예측보다 앞서가면 fallback → end_y = col_y → 검증 실패
3. **line 1484** `s.line_spacing.max(0)` — 음수 trailing ls 보정 거부
4. **line 1536** `MAX_BACKWARD_PX = 8.0` — 누적 drift > 8px 회복 불가

각각 따로 보면 정당한 가드(과거 회귀 보호)지만, 조합되어 forward drift 회수 경로가 모두 막힌 상태.

### 의미 혼재

`compute_line_spacing_hwp` (line_breaking.rs:807) 의 `.max(0.0)` clamp 는 reflow 경로(편집 시) 에만 적용되며 초기 IR ls 는 음수를 유지한다. 따라서 IR 자체는 음수 ls 를 보존하나 layout advance 가 이를 활용하지 못하는 구조.

## 3. 정정 방향 (가설 — 하이브리드 C 안)

기존 분석에서 후보 A(source fix) / B(recovery fix) 검토. 단독 적용은 회귀 또는 시각 점프 우려가 있어 **두 축의 좁은 부분만 결합한 하이브리드 C** 채택.

### C-1. Source fix (좁은 적용)

**대상**: TAC 표 호스트 advance 가 outer_margin+table_h+margin 만 사용하여 ls 를 무시하는 지점.

**가설 H1**: TAC 표 호스트 layout 분기에서 음수 ls 를 advance 에 가산.

**대상 아님(범위 제외)**:
- 일반 paragraph 의 음수 ls 처리 — 현재도 `y += line_height + line_spacing_px` (paragraph_layout.rs:2657) 가 음수 ls 를 가산하므로 이론상 동작. 빈 문단 advance 가 실측 lh 만으로 보이는 이유는 별도 조사(Stage 2) 필요. 만약 composer 가 음수 ls 를 보존하지 않는 경로가 있다면 거기서 정정.

### C-2. Recovery fix (보강)

**대상**: VPOS_CORR 가 forward drift 를 회수하지 못하는 4중 가드.

**가설 H2-1**: `lazy_base < 0` fallback 을 `lazy_base.max(0)` 로 변경. y_offset 이 vpos 예측보다 앞서가면 절대 vpos 좌표(`col_area.y + vpos_end*scale`) 사용.

**가설 H2-2**: `trailing_ls_hu = s.line_spacing.max(0)` 의 `max(0)` 제거(또는 음수 허용). 음수 ls 를 그대로 trailing 보정에 반영.

**가설 H2-3 (조건부)**: 가설 H1 으로 source 가 정정되면 누적 drift 가 ≤ MAX_BACKWARD_PX 이내로 수렴할 수 있다. 그렇다면 line 1402 가드 약화 / MAX_BACKWARD_PX 확장은 **불필요**. Stage 2 측정 후 결정.

### 적용 순서 (점진적)

1. C-2 (H2-1, H2-2) 우선 적용 — 영향 범위 좁고 가산 로직만 변경.
2. C-1 (H1) 적용 — TAC 표 호스트 advance 정정.
3. 회귀 검증 후 H2-3 (가드 약화) 필요 여부 판단.

- `compute_line_spacing_hwp` (line_breaking.rs) 의 `.max(0.0)` clamp 변경 — reflow 경로 변경은 본 결함과 무관, 추후 별도 검토

- HWPX 파일 동등 검증 — hongbo 샘플은 .hwp 이므로 우선 HWP5 경로 정정. HWPX 영향은 IR 동일하므로 자연스럽게 적용됨
