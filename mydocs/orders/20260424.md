## 1. Task #265 — HWP 3.0 파일 감지 + 친절한 에러 메시지

### 배경

@jangster77 님의 제보 (2026-04-23). `samples/issue_265.hwp` 를 rhwp 의 어떤 경로에서도 열지 못함. 원인: **HWP 3.0** 바이너리 포맷 (1996~1998년 경 한글 3.0). 현재 rhwp 는 HWP 5.0 + HWPX 만 지원.

### 처리 방침
작업지시자 지시: "HWP 3.0 정식 지원은 이번 마일스톤 범위 아님. 감지 + 친절한 에러 메시지까지만".

### 변경 범위
- **코드** `src/parser/mod.rs`:
  - `FileFormat::Hwp3` variant + `detect_format` 에 17바이트 프리픽스 체크
  - `ParseError::UnsupportedFormat { format, hint }` + `Display` 확장
  - `parse_document` 에서 Hwp3 → 한글 안내 메시지 반환
- **코드** `src/error.rs` — **발견된 추가 버그 수정**:
  - `From<ParseError> for HwpError` 등 3개 `From` 이 `{:?}` (Debug) 를 쓰고 있어 친절한 Display 메시지가 사용자에게 도달하지 못함. `{e}` (Display) 로 전환. CFB 매직 바이트 내부 누출도 함께 해결
- **코드** `rhwp-studio/src/main.ts` — **UI 전달 누락 수정**:
  - 기존: 상태 표시줄 (22px, `overflow: hidden`) 에만 쓰고 있어 긴 메시지 대부분 가려짐
  - 수정: `showLoadError` 유틸 + 우상단 토스트 (`showToast` 기존 유틸 재사용). 3 로드 경로 (`loadFile` · `loadFromUrlParam` · `open-document-bytes` 이벤트) 통합
  - `open-document-bytes` 이벤트 핸들러가 WASM throw 를 catch 하지 않아 "파일 열기" 커맨드 에선 에러가 완전 묵살되던 문제도 함께 수정
- **테스트** 6건 (detect + parse + Display 회귀 방어)
- **WASM 재빌드**: 4,076,166 → 4,081,046 bytes (+4,880 bytes)

## 2. Task #275 — WASM canvas 렌더러 OLE RawSvg/Placeholder 처리 복구

### 배경

작업지시자 제보: `samples/bitmap.hwp`, `samples/한셀OLE.hwp` 를 rhwp-studio 에서 열면 파일 로드는 성공하나 본문이 **완전히 빈 페이지**. 네이티브 CLI SVG 내보내기는 정상.

### 처리 방침
3단계 하이퍼워터폴. 이슈→브랜치→수행계획→구현계획→단계별 구현 (Placeholder arm → A 경로 → B 경로 + 검증) → 최종 보고서. 각 단계별 승인 요청.

### 근본 원인

`src/renderer/layout/shape_layout.rs:983-1094` `ShapeObject::Ole` 처리가 OLE 컨테이너 내부에서 네이티브 이미지/EMF/OOXML 차트를 추출해 `RenderNodeType::RawSvg` 로, 추출 실패 시 `RenderNodeType::Placeholder` 로 래핑함. **`svg.rs` 는 두 노드 타입을 처리하지만 `web_canvas.rs` 는 둘 다 match arm 이 없어 `_ =>` 로 빠져 암묵 무시됨** → 빈 페이지.

### 변경 범위
- **코드** `src/renderer/svg_fragment.rs` (신규, +247줄):
  - `find_svg_attr_value`, `try_parse_single_image_data_url`, `decode_base64_data_url`, `is_svg_prefix`, `wrap_svg_fragment` 5 헬퍼
  - 단위 테스트 19건
  - web_canvas (wasm32 전용) 가 아닌 렌더러 공용 모듈로 분리 → 네이티브 테스트 가능
- **코드** `src/renderer/web_canvas.rs`:
  - `RenderNodeType::Placeholder` arm (+26줄) — svg.rs 와 동등 출력 (회색 배경 + 점선 테두리 + 중앙 라벨)
  - `RenderNodeType::RawSvg` arm (+25줄) — A 경로 (`<image data:...>` 단일 요소 직접 디코드) + B 경로 (`<g>...</g>` 복합 SVG 를 `<svg>` 로 래핑 후 draw_image 경유 async 로드)
  - `detect_image_mime_type` 에 SVG 분기 추가 → 별도 `draw_svg` 함수 없이 기존 draw_image 파이프라인 재사용
- **WASM 재빌드**: 4,043,989 → 4,051,019 bytes (+7,030 bytes)

### 검증
- `cargo test --lib svg_fragment`: **19 passed / 0 failed**
- `cargo test --lib`: **968 passed / 14 failed** (+19 신규, baseline 14 실패는 사전 존재, 변화 없음)
- `cargo check --lib --target wasm32-unknown-unknown`: clean
- `wasm-pack build --target web`: 성공
- `npx tsc --noEmit` (rhwp-studio): clean
- E2E (puppeteer + headless Chrome):
  - `bitmap.hwp` 빈 페이지 → **비트맵 손글씨 이미지 정상 렌더**
  - `한셀OLE.hwp` 빈 페이지 → **노란 스프레드시트 이미지 정상 렌더**
  - B 경로 (`<g><rect/><image/><text/></g>` 강제 재현 → 원복): 복합 SVG 전체 렌더 확인
  - Placeholder (FORCE_PLACEHOLDER 가드 → 원복): 점선 테두리 + 중앙 라벨 확인
  - 회귀: `biz_plan.hwp`, `form-002.hwpx` 변화 없음

### 핵심 설계 결정
- **헬퍼 모듈 분리**: `svg_fragment` 를 wasm32 gate 없이 등록 → 네이티브 단위 테스트 확보. web_canvas 는 wasm32-only 이라 내부에 두면 테스트 불가
- **draw_image 재사용**: SVG 전용 `draw_svg_async` 분리 대신 MIME 감지 한 분기로 통합 → IMAGE_CACHE / HtmlImageElement async / 재렌더 로직 중복 방지
- **viewBox = bbox**: 조각 내부의 페이지 절대좌표 = drawImage 위치 (1:1 단순 투영) — 좌표 변환 로직 제거로 디버깅 용이

### 교훈
- **표면 증상 "빈 페이지"에서 뒤로 3단계** (canvas 렌더러 부재 → RawSvg/Placeholder 노드 타입 → svg.rs/web_canvas.rs 비대칭) 추적이 필요한 버그. 증상만 보면 "파서 버그" 로 오진하기 쉬운데 `rhwp export-svg` (CLI) 가 정상 동작한다는 교차 검증으로 빠르게 렌더 경로로 좁힘
- **두 렌더러 간 match arm 대칭성 검사**가 자동화되어 있지 않음 → 후속: `svg.rs` 와 `web_canvas.rs` 의 `RenderNodeType` match arm 커버리지 비교 테스트 고려 (M101+ 후보)
- **강제 재현 기법**: 프로덕션 샘플이 없는 경로 (B 경로, Placeholder) 도 shape_layout.rs 에 임시 가드를 한 줄 넣어 WASM 재빌드 → 스크린샷 → 원복 사이클로 시각 검증 가능. diff clean 상태로 원복 확인이 핵심
