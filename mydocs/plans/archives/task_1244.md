## 문제 요약

rhwp studio에서 탭 문자를 삽입한 뒤 HWP로 저장하면, 한컴 편집기에서 탭이 모두 사라진다.

## 근본 원인

HWP5 바이너리에서 탭 문자(0x0009)는 7개의 u16 확장 데이터(`tab_extended[i]: [u16; 7]`)를 동반한다.
`ext[6]`에는 항상 `0x0009` 마커가 있어야 한컴이 탭으로 인식한다.

| 경로 | 동작 |
|------|------|
| HWP 파일 파싱 | `tab_extended` 올바르게 채워짐 |
| 편집으로 탭 삽입 | `document_core`가 `tab_extended` 전혀 관리하지 않음 → `Vec::new()` 상태 유지 |
| HWP 직렬화 (`body_text.rs:510-512`) | `tab_extended`가 비어있으면 `[0,0,0,0,0,0,0]` 출력 → `ext[6]=0` (마커 없음) |
| 한컴 편집기 | `ext[6]≠0x0009`이므로 탭으로 인식 못함 → 탭 소멸 |

## 수정 방향

`body_text.rs`의 폴백 경로에서 `ext[6]`를 `0x0009`로 채워 한컴이 탭을 인식할 수 있게 한다.

- `ext[0]`(탭 폭)은 `0`으로 두면 한컴이 TabDef(탭 정지 목록)에서 재계산한다.
- `ext[2]`(type/leader)도 `0`(LEFT/채움 없음)으로 두면 기본 탭으로 동작한다.
- 따라서 폴백 데이터: `[0, 0, 0, 0, 0, 0, 0x0009]`

## 수정 범위

- `src/serializer/body_text.rs` — 폴백 `[0; 7]` → `[0, 0, 0, 0, 0, 0, 0x0009]`

`document_core`에서 삽입 시점에 `tab_extended`를 채우는 방식도 검토하나, 이슈 범위는 위 단일 수정으로 충분하다.

## 검증 계획

1. rhwp studio에서 새 문서 → 탭 삽입 → HWP 저장
2. 한컴 편집기에서 열어 탭이 정상 표시되는지 확인
3. `cargo test` 통과 확인
