# 수행 계획서 — Task M100-1341: rhwp-studio PWA OS 파일 연결 지원

- 이슈: https://github.com/edwardkim/rhwp/issues/1341
- 제목: PWA 가 OS 파일연결을 지원하게 해주십시오
- 작성일: 2026-06-09
- 브랜치: `local/task_m100_1341`

## 1. 문제 요약

현재 `https://edwardkim.github.io/rhwp/` 로 배포되는 rhwp-studio PWA는
OS 파일 연결을 선언하지 않는다. 따라서 사용자가 `.hwp` / `.hwpx` 파일을
더블클릭하거나 "다음으로 열기"에서 설치된 PWA를 선택해 바로 열 수 없다.

요청 의도는 HOP 같은 별도 데스크톱 래퍼 없이도 설치된 PWA가 문서 파일을 직접
열 수 있게 하는 것이다.

## 2. 현황 조사

현재 확인한 코드 기준:

| 영역 | 위치 | 관찰 |
|---|---|---|
| PWA manifest | `rhwp-studio/vite.config.ts` | `vite-plugin-pwa` 사용, `start_url=/rhwp/`, `scope=/rhwp/`, `file_handlers` 없음 |
| 파일 열기 | `rhwp-studio/src/command/file-system-access.ts` | `FileSystemFileHandleLike`, `readFileFromHandle()` 존재 |
| 일반 파일 입력 | `rhwp-studio/src/main.ts` | `.hwp/.hwpx` 확장자 검사 후 `loadFile()` |
| 공통 로딩 | `rhwp-studio/src/main.ts` | `open-document-bytes` 이벤트에서 미저장 변경 확인, `loadBytes()`, `showLoadError()` 처리 |
| 저장 | `rhwp-studio/src/command/commands/file.ts` | 사용자 `저장`/`다른 이름으로 저장` 명령 시에만 파일 핸들에 write |

공식 문서 기준으로 PWA 파일 연결은 두 부분으로 나뉜다.

1. Web App Manifest `file_handlers`로 처리 가능한 파일 확장자/MIME을 선언한다.
2. 앱 JavaScript의 `window.launchQueue.setConsumer()`에서 OS launch로 전달된
   파일 핸들을 받아 실제 로딩을 수행한다.

## 3. 목표

- 설치된 rhwp-studio PWA가 `.hwp`, `.hwpx` 파일을 OS "열기 프로그램" 후보로
  등록할 수 있게 한다.
- OS launch로 들어온 파일을 기존 문서 로딩 흐름으로 열어 UI/오류 처리 일관성을
  유지한다.
- 로드 실패 시 기존 사용자 알림 UI(`showLoadError`)를 사용하고, 잘못된 파일
  핸들이 저장 경로에 남지 않도록 한다.
- 저장 동작은 기존처럼 사용자 명시 액션에만 연결한다.
- 기존 한컴오피스/한컴뷰어 설치 사용자의 기본 파일 연결을 침범하지 않는지
  수동 검증 항목으로 포함한다.

## 4. 비목표

- OS 기본 프로그램을 rhwp-studio로 자동 변경하지 않는다.
- 미설치 브라우저 탭에서 OS 더블클릭 파일 열기를 지원하려 하지 않는다.
- Safari/Firefox까지 동일 기능을 보장하지 않는다.
- HOP 또는 네이티브 래퍼 기능 전체를 대체하지 않는다.
- `.hwp/.hwpx` 외 확장자나 넓은 binary MIME을 등록하지 않는다.

## 5. 보안 및 UX 검토

### 5.1 권한 모델

File Handling API는 설치된 PWA에 한해 OS 파일 핸들러 등록을 허용한다. 파일 열기
시 브라우저/OS의 사용자 선택과 권한 확인이 개입한다. 따라서 단순 웹사이트 방문만으로
임의 로컬 파일을 읽는 구조는 아니다.

### 5.2 rhwp 적용 시 가드

- `file_handlers.accept`는 `.hwp`, `.hwpx` 전용 MIME만 사용한다.
- `application/octet-stream`, `*/*`, `application/*` 같은 넓은 매칭은 금지한다.
- launch로 받은 파일도 기존 파일 입력과 같은 확장자 검사를 거친다.
- 파일 바이트 로딩 후 HWP/HWPX 시그니처 검증 또는 WASM 파서 실패 시
  기존 오류 UI로 종료한다.
- `currentFileHandle`은 로드 성공 후에만 설정한다.
- 자동 저장/자동 덮어쓰기는 추가하지 않는다.
- 파일 핸들은 IndexedDB/localStorage/Service Worker cache에 저장하지 않는다.

### 5.3 기존 한컴 설치 사용자 영향

가장 중요한 UX 리스크는 기존 `.hwp/.hwpx` 기본 앱이 한컴오피스/한컴뷰어로 잡힌
사용자다. PWA `file_handlers` 등록 후 rhwp-studio가 "열기 프로그램" 후보로
추가되며, OS/브라우저 정책에 따라 기본 핸들러가 없는 환경에서는 기본 후보처럼
승격될 수 있다.

따라서 다음을 검증한다.

- 한컴 설치 상태에서 `.hwp/.hwpx` 기본 연결이 유지되는지 확인한다.
- "다음으로 열기" 후보에 rhwp-studio가 추가되는 정도인지 확인한다.
- 한컴 미설치 상태에서 rhwp-studio가 기본 후보로 올라오는지 확인한다.
- 앱 내부에 "기본 앱으로 설정" 유도 문구를 추가하지 않는다.

## 7. 검증 계획

자동 검증:

- `cd rhwp-studio && npm run build`
- `cd rhwp-studio && node --experimental-strip-types --test tests/*.test.ts`
- 필요 시 PWA manifest 산출물에 `file_handlers`가 포함되는지 정적 확인

수동 검증:

- Chrome 또는 Edge에서 rhwp-studio PWA 설치
- `.hwp` 파일 "다음으로 열기"에서 rhwp-studio 후보 노출 확인
- `.hwpx` 파일 "다음으로 열기"에서 rhwp-studio 후보 노출 확인
- 파일 열기 허용 후 기존 문서 로딩 UI로 열리는지 확인
- 한컴 설치 환경에서 기존 기본 연결이 임의 변경되지 않는지 확인
- 한컴 미설치 환경에서 기본 연결 승격 여부 확인

권장:

- `docker compose --env-file .env.docker run --rm wasm`
- 작업지시자 macOS/Windows 실제 OS 파일 연결 판정
